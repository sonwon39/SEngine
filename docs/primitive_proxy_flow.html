3. 핵심 차이 요약

항목 현재 SEngine 언리얼
Proxy의 역할 draw call을 직접 호출 데이터 mirror, FMeshBatch만 채워서 넘김
스레드 모델 단일 게임 ↔ 렌더 스레드 분리 (ENQUEUE_RENDER_COMMAND 경계)
per-primitive CB (아직 없음) — root descriptor table 슬롯에 직접 bind 예정 GPUScene이라는 StructuredBuffer에 모두 모음, 셰이더가 PrimitiveId로 lookup
View 처리 없음 — 모든 proxy 무조건 그림 GetViewRelevance로 패스별로 분류
draw 발행 주체 PrimitiveProxy 자신 FMeshPassProcessor (Proxy 외부)

4. 점진적으로 언리얼 방향으로 가는 단계

Step 0 (현재)
Proxy가 buffer view + heap + handle 들고 직접 draw
↓ CB 추가만 한다면 — 여기서 멈춤
Step 1 — MVP
Proxy에 per-primitive CB 추가, RootDescriptorTable에 b1 슬롯 등록
↑ 지금 질문한 방향. "언리얼 비슷"은 아님
↓
Step 2 — Draw 발행 분리
Proxy::Render() 제거. 대신 FillMeshBatch(...) 같은 메서드로 구조체만 채움
RenderEngine이 MeshBatch 리스트를 모아서 일괄 draw
↓
Step 3 — GPUScene 흉내
per-draw CB 폐기 → StructuredBuffer<PrimitiveData> 한 개
셰이더에서 PrimitiveId로 lookup. RootDescriptorTable 슬롯 수가 줄어듬
↓
Step 4 — View Relevance
opaque / translucent / depth-only 패스별 필터링

결론: Step 1만 추가해도 엔진 작동에는 충분하지만, "언리얼과 비슷"하다고 부르려면 적어도 Step 2 (draw 발행 분리)까지는 가야 함.
